# Checkpoint pre-training baseline computation at per-candidate granularity, persisting on-demand to a separate W&B artifact and resuming via the existing job chain

`Baseline.generate_baseline_data` runs three sequential sweeps (median, dynamic, constant), each evaluating ~20 candidate configs × 10 inner DP-SGD trainings plus an 8-rep final phase — ~620 inner trainings in one blocking call before the main loop starts. When a SLURM job times out here the whole run dies and the resubmitted job re-runs every sweep from scratch, because the existing job-chain machinery (`util/job_chain.py`, `save_checkpoint`) is wired only into the **main training loop**, not the baseline phase.

We make the **pre-training baseline path** (`baseline_log_interval > 0`) resumable by piggybacking on the existing job chain. The candidate loops check `time_limit_approaching()` / `shutdown_requested()` at every candidate boundary; on trigger they raise `BaselineInterrupted` carrying the in-progress sweep state. `generate_baseline_data` catches it, attaches the already-completed sweep dataframes, and re-raises to `main.py`, whose pre-training block does save-progress → `run.finish()` → `resubmit_if_requested()` → `return`. The partial blob — `{completed: {name: df}, in_progress: {sweep_name, candidate_idx, best_params (floats), best_acc, key (np)}}` — is written **on-demand only** (when the time limit fires) to a **separate** `baseline-progress-{run_id}` W&B artifact (local pickle + W&B fallback, alias `latest`), distinct from the complete `baseline-{run_id}` artifact, which is still written only on full completion. On restart, `main.py` tries the complete-baseline restore first (reuse, as today), then the progress artifact (pass `resume_state` into `generate_baseline_data`), then starts fresh.

## Why

- **Per-candidate, not per-sweep.** A single sweep (~208 inner trainings) can itself exceed the wall clock, so checkpointing only at sweep boundaries (3 units) would not prevent the crash. Per-candidate bounds redo to ~10 inner trainings and survives the one-sweep-doesn't-fit case. The 8-rep final phase is treated atomically (a single boundary check before it starts, encoded as `candidate_idx == num_candidates` ⇒ "candidates done, reps pending") because its cost is comparable to one candidate and per-rep df-append state would add complexity for ~8 trainings of savings.
- **On-demand-only writes.** Mid-sweep state is tiny and baselines are recomputable, so we persist only when the graceful `SLURM_JOB_END_TIME` path warns us, rather than after every candidate (≈60 chatty uploads) or every N. This deliberately gives up durability against an **ungraceful hard kill** (OOM, node failure) — that case simply recomputes the baseline.
- **Separate progress artifact.** The complete `baseline-{run_id}` artifact is consumed downstream by `compile_results_fetch._baseline_means` and `dp_psac_ref`, which expect a complete df with required columns. Keeping partial state in its own artifact means a partial df is never visible to those consumers.
- **Must upload to W&B.** `_ckpt_dir` uses `$SLURM_TMPDIR` when set, which is node-local scratch wiped at job end, so a local-only partial blob would not survive a resubmit (a *new* SLURM job). This is the same reason `save_checkpoint` uploads.
- **No Orbax for partial state.** Every baseline schedule reconstructs from a handful of sampled scalars (`c_0`/`eta_C`; `rho_mu`/`rho_c`/`c_0`; `sigma`/`clip`) with `privacy_params` re-injected from `self.privacy_params`. Orbax was only needed for the *completed* dynamic schedule, which mid-sweep is just its three floats — so the partial blob is a plain pickle.
- **Determinism is preserved for free.** Saving the in-progress sweep's loop key + candidate index gives exact continuation of that sweep; on resume, the inter-sweep `jr.split` chain is replayed cheaply (key derivation is deterministic and inexpensive — only the DP-SGD trainings are costly), so not-yet-started sweeps draw identical candidates.

## Considered and rejected

- **Per-sweep granularity (3 checkpoint units).** Rejected: a single sweep can exceed the wall clock, so this would not stop the crash it is meant to prevent.
- **Periodic / every-candidate writes** (mirror the main loop's `checkpoint_every`). Rejected as the default: insures against hard kills but adds upload chatter for state that is tiny and fully recomputable; the timeout (not hard kill) is the stated protection target.
- **Extend the existing `baseline-{run_id}` artifact with a `complete` flag.** Rejected: every downstream consumer (`_baseline_means`, `dp_psac_ref`) would have to learn to check the flag or risk reading a partial df.
- **Return-value sentinel instead of an exception.** Rejected: the candidate loop is two calls deep and several methods return tuples already; threading an optional "not finished" object through every return path is noisier than one `try/except` at the `main.py` call site.
- **Hardening the end-only path (`baseline_log_interval == 0`) too.** Out of scope: the real experiments (`create_experiments.py`) set `baseline_log_interval=200`, so they take the pre-training path. The shared `generate_baseline_data` is gated by a `resumable: bool = False` parameter (the end-only caller keeps `False`), leaving that path byte-for-byte unchanged.

## Consequence

- **An ungraceful kill loses baseline progress.** Resume works only after a graceful `SLURM_JOB_END_TIME`-triggered save; an OOM/node failure mid-baseline leaves no progress artifact and the next job recomputes from scratch. Fail-safe but not crash-proof.
- **`generate_baseline_data` gains a `resumable` flag and a `resume_state` input,** and the candidate loops in `baseline_sweep` / `_constant_schedule_sweep` gain start-index/best-so-far/key resume arguments and a per-candidate time-limit check.
- **`main.py`'s pre-training baseline block gains a `try/except BaselineInterrupted`** that mirrors the post-loop resubmit dance, and a progress-restore step ahead of the existing complete-baseline restore.
- **A new `baseline-progress-{run_id}` artifact namespace exists** alongside `baseline-{run_id}` and `checkpoint-{run_id}`; it is superseded (and can be ignored) once the complete baseline is written.
- **Baseline-phase resubmits reuse the same W&B run id** (`init_wandb_run` resumes with `checkpoint_run_id` set and `checkpoint_step=None`), so progress artifacts keyed by `run_id` are found under the same id across the chain.
